## CRITICAL

---

**Severity:** CRITICAL
**Dimension:** Security and Access
**Location:** `src/internal/common/store_notification.go:45–77`
**Violation:** `fmt.Sprintf` injects an `orderDirection` string literal into a raw SQL query. The value is derived from a boolean (`isForward := options.MinID > 0`) and can only be `"ASC"` or `"DESC"` today, but the pattern itself establishes a precedent of SQL string construction. Any future developer extending this query (e.g., adding a user-selectable sort field) may imitate this pattern without understanding the injection boundary.
**Mandated Refactor:** Use a Go-level `if/else` to select between two fully static query strings. Never interpolate any value — even an internal one — into SQL:

```go
var query string
if options.MinID > 0 {
    query = `SELECT ... ORDER BY n.id ASC LIMIT $4`
} else {
    query = `SELECT ... ORDER BY n.id DESC LIMIT $4`
}
rows, err := s.db.QueryContext(ctx, query, userID, options.MinID, options.MaxID, targetCount)
```

**Severity:** CRITICAL
**Dimension:** Security and Access
**Location:** `src/internal/common/control.go:328–347` (`ListObjectTypeCounts`)
**Violation:** `fmt.Sprintf` builds a `SELECT COUNT(*) FROM "%s"` query by interpolating a table name from `objectTables`. An `allowedTable()` guard exists, but the pattern normalizes SQL construction via string interpolation. If the allowlist or the guard call is ever removed or refactored around, this becomes a SQL injection vector.
**Mandated Refactor:** Hardcode one query per table as a named constant, or use a `switch` statement mapping each allowed table to its static query string. Eliminate the `fmt.Sprintf` SQL construction entirely. The comment acknowledging the risk is insufficient mitigation.

## WARNING

---

**Severity:** WARNING
**Dimension:** Structural Modularity / Single Responsibility
**Location:** `src/cmd/agent/run.go:33–168` (`runAgentOnce`)
**Violation:** Single function exceeds 130 executable lines and performs six distinct responsibilities: (1) state loading and validation, (2) token refresh and persistence, (3) ticket polling, (4) ack-comment posting, (5) workspace preparation, (6) LLM loop dispatch and result handling. This violates SRP — a state-refresh failure, a workspace error, and a comment-posting failure all live in the same function.
**Mandated Refactor:** Extract into composable, testable units:
- `ensureFreshToken(state) (AgentState, error)` — refresh logic only
- `acquireNextTicket(client) (AssignedTicket, bool, error)` — polling only
- `processTicket(client, cfg, state, ticket) error` — orchestrate workspace + LLM + comments

**Severity:** WARNING

**Location:** `src/cmd/agent/llm.go:167–189` (`runLMStudioTicketLoop`)
**Violation:** The `case "", "stop":` and `default:` arms of the `switch choice.FinishReason` block are byte-for-byte identical — same answer extraction, same `OnAssistant` callback, same return. This creates a maintenance trap: a future bug fix applied to one branch is silently not applied to the other.

**Severity:** WARNING
**Dimension:** Performance & Complexity
**Location:** `src/internal/common/store_user.go:241–284` (`ensureUserActorIdentityBackfill`)
**Violation:** Issues one `BEGIN/COMMIT` transaction per missing user in a loop. For a deployment with N users lacking actor identities, this produces N round-trips. The function runs at server startup, so on a degraded or new database this blocks the server from accepting connections until all N transactions complete.
**Mandated Refactor:** Collect all missing users first (as the code already does), then provision them in a single transaction:

```go
tx, err := s.db.BeginTx(ctx, nil)
if err != nil { return err }
defer tx.Rollback()
for _, u := range missing {
    if err := provisionUserActorTx(ctx, tx, u.id, u.username, s.baseURL); err != nil {
        return err
    }
}
return tx.Commit()
```

**Severity:** WARNING
**Dimension:** Performance & Complexity
**Location:** `src/cmd/web/data.go:452–469` (`renderRepoPageBySlug`)
**Violation:** Double-fetch on the assigned tracker. The function first fetches all local trackers via `ListTicketTrackers`, searches the resulting slice for `repo.TicketTrackerSlug`, and if not found, fires a second DB call (`GetTicketTrackerBySlug`). Since `ListTicketTrackers` queries all `is_local = TRUE` rows, a tracker that exists locally will always appear in the list. The fallback is dead logic that adds latency in non-existent edge cases while hiding a potential data inconsistency.
**Mandated Refactor:** Remove the fallback second query. If the assigned tracker is not in the `ListTicketTrackers` result, the data is inconsistent and should be surfaced as an error or treated as unassigned — not silently hidden by an extra round-trip:

```go
var assigned *common.TicketTracker
if repo.TicketTrackerSlug != "" {
    for i := range trackers {
        if trackers[i].Slug == repo.TicketTrackerSlug {
            assigned = &trackers[i]
            break
        }
    }
    // If still nil: tracker is assigned but not local — treat as unassigned
}
```

**Severity:** WARNING
**Dimension:** Performance & Complexity
**Location:** `src/cmd/web/login.go:104–119` (`requireAuth`)
**Violation:** Every authenticated HTTP request performs a `store.UserExists` database query before the handler is invoked. In a system with session management, the session token is already a cryptographic proof of authentication. The `UserExists` call adds a synchronous DB round-trip to every page load, providing marginal value (detecting deleted users between requests) at steady-state cost proportional to request volume.
**Mandated Refactor:** Eliminate the per-request `UserExists` call. Handle user deletion via session invalidation at deletion time (`store.DeleteUser` should also call `sessionManager.Clear(userID)` or rotate the session secret). If the "ghost session" scenario must be guarded, consider caching the check with a short TTL rather than querying on every request.

**Severity:** WARNING
**Dimension:** Architectural Alignment
**Location:** `src/internal/common/control.go` — `buildLocal*` family of functions
**Violation:** `buildLocalRepositoryObject`, `buildLocalTicketObject`, `buildLocalTicketCommentObject`, etc. all return `map[string]any`. This uses Go's dynamic typing to represent domain objects that have well-known, stable ActivityPub schema shapes. There is no compile-time validation that required fields (e.g., `"id"`, `"type"`, `"context"`) are present.
**Mandated Refactor:** Define typed structs (or interfaces) for each ActivityPub object type and serialize to JSON only at persistence time. This enforces shape correctness at compile time and makes the domain model self-documenting.

## NITPICK

---

**Severity:** NITPICK
**Dimension:** Cognitive Readability
**Location:** `src/cmd/web/api_v1.go` — multiple handlers; `src/cmd/web/data.go` — multiple handlers
**Violation:** Approximately half the route handlers manually check `if r.Method != http.MethodGet { writeJSONError(...) }` even though the route is already registered with a method-prefixed pattern (`"GET /api/v1/..."`) that Go's `net/http` ServeMux (1.22+) will enforce. The manual check is unreachable dead code for those routes. For the routes registered without a method prefix (e.g., `mux.HandleFunc("/web/login", app.login)`), the check is necessary.
**Mandated Refactor:** Standardize all routes to use method-prefixed patterns (`"GET /path"`, `"POST /path"`) and remove the manual method check from handlers registered with explicit method patterns. Reserve the `switch r.Method` dispatch only for routes intentionally handling multiple methods.

**Dimension:** Structural Modularity
**Location:** `src/cmd/web/main.go:36–37`
**Violation:** `homeTmpl` and `loginTmpl` are package-level `var` globals initialized at package init time. While harmless in a single-binary deployment, mutable package-level state makes unit testing non-trivial and violates the explicit dependency injection present elsewhere in the codebase (the `application` struct uses field injection for all other dependencies).
**Mandated Refactor:** Move template initialization into `newHandler` and attach them to the `application` struct:

```go
type application struct {
    ...
    templates struct {
        home  *template.Template
        login *template.Template
    }
}
```

**Severity:** NITPICK
**Dimension:** Documentation & Intent
**Location:** `src/cmd/web/login.go:38`, `57`
**Violation:** `app.loginRateLimiter` is checked for `nil` at two call sites (`if app.loginRateLimiter != nil && ...`), but `newLoginRateLimiter` is always called unconditionally in `newHandler`. This nil-guard implies the field could be absent, creating ambiguity about the invariant. Readers must trace the construction path to understand whether nil is actually reachable.
**Mandated Refactor:** Remove nil guards and rely on the invariant enforced at construction time, or introduce a no-op `loginRateLimiter` implementation that always returns `true` for `Allow()` — making the nil-as-disabled-state explicit through the Null Object pattern.
